How I work

The portfolio is the output of a consistent five-step methodology. Each tool in the portfolio instruments one or more of these steps.

Step 1
Research

Investigate a component against WAI-ARIA Authoring Practices, screen-reader behavior, and WCAG criteria before proposing a contract.

Step 2
Guideline

Document the accessible contract: roles, states, keyboard interaction, focus management, and announcement expectations.

Step 3
Audit

Layered testing: axe, Vitest + Testing Library, Playwright, plus manual screen-reader verification.

Step 4
Triage

Every violation gets a WCAG criterion, reproduction steps, expected vs. actual, and acceptance criteria before it reaches a developer.

Step 5
Fix & regression

Ship the fix with tests that lock in the accessible behavior so the regression is caught automatically next time.

Architecture

Why the portfolio is shaped the way it is.

Problem

Accessibility knowledge is scattered

Patterns, heuristics, test recipes, and ticket templates live across people's heads, wikis, and ad-hoc scripts. AI tools don't consistently apply them.

Solution

Composable accessibility modules

Small, focused repos — each handling one job (audit, simulation, AI guidance, ticketing) — unified by a shared portfolio surface and a consistent methodology.

Scope

Link first, merge later

This repo is the umbrella and public entry point. Linked modules stay in their own repos until there's a real benefit to merging — shared release flow, shared tests, or reduced duplication.
