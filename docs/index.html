02 / Method

The five-step pattern behind every repo.

01
Research

Investigate a component against WAI-ARIA Authoring Practices, screen-reader behaviour, and WCAG criteria before proposing a contract.

02
Guideline

Document the accessible contract — roles, states, keyboard interaction, focus management, announcement expectations.

03
Audit

Layered testing — axe, Vitest with Testing Library, Playwright, plus manual screen-reader verification.

04
Triage

Every violation gets a WCAG criterion, reproduction steps, expected vs. actual, and acceptance criteria — before it reaches a developer.

05
Fix & regression

Ship the fix with tests that lock in the accessible behaviour, so the regression is caught automatically next time.

03 / Architecture

Why the portfolio is shaped this way.

Problem

Accessibility knowledge is scattered.

Patterns, heuristics, test recipes, and ticket templates live across people's heads, wikis, and ad-hoc scripts. AI tools don't consistently apply them.

Solution

Composable modules.

Small, focused repos — each handling one job (audit, simulation, AI guidance, ticketing) — unified by a shared portfolio surface and a consistent methodology.

Scope

Link first, merge later.

This repo is the umbrella and public entry point. Linked modules stay in their own repos until there's a real benefit to merging — shared release flow, shared tests, or reduced duplication.
